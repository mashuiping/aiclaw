# AIClaw 落地挑战

本文档记录 AIClaw 项目从「命令路由器」进化为「AI 运维专家」面临的核心挑战。

## 项目当前定位

**当前状态**：基于正则匹配的意图解析 + 命令路由，执行后直接返回原始输出。

**目标状态**：能理解自然语言查询、自动规划排查步骤、LLM 综合分析、给出根因和建议。

---

## P0 - 不解决不能落地

### 1. 意图理解能力天花板极低

**现状**：
- 纯正则匹配（`intent.rs`），依赖枚举式 keyword matching
- 无法理解语义相近但表达不同的问法
- 置信度计算是简单的 keyword 加分

**影响**：用户必须用机器人能理解的「机器语言」提问，而非自然表达。

**解决方向**：
- 集成 LLM 做意图识别和实体抽取
- 或使用 embedding-based 相似度匹配

---

### 2. 缺乏 AI 推理能力 — 执行完即结束

**现状**：
```rust
// orchestrator.rs - 直接返回原始输出
output += out;
```
- 执行完命令就结束，不对结果做处理
- 用户问"为什么 Pod 挂了"，返回的是原始 `kubectl describe` 输出

**影响**：用户仍需自己分析原始数据，无法得到「结论」。

**解决方向**：
- 构建 Plan-Execute-Reason 模式
- 所有结果过 LLM 总结层
- 自动关联 Logs + Metrics + Events 综合分析

---

### 3. 安全机制薄弱

**现状**：
- Shell 执行使用黑名单机制（`dangerous_commands`）
- 黑名单本质不安全
- 没有操作审计

**影响**：生产环境无法接受任意 shell 执行。

**解决方向**：
- 白名单机制（只允许预批准的 kubectl 命令模式）
- 敏感操作二次确认
- 完整审计日志

---

## P1 - 影响核心体验

### 4. 对话体验几乎为零

**现状**：
- Session 只做 interaction 记录，无上下文管理
- 用户不能追问："然后呢？"、"能详细解释一下吗？"
- 每次对话独立 start from scratch

**影响**：无法进行多轮故障排查。

**解决方向**：
- 多轮对话上下文管理
- 支持追问和澄清
- 对话式流程引导

---

### 5. 多集群支持空白

**现状**：
- 配置中支持多集群，但运行时没有集群切换机制
- 没有跨集群关联分析

**影响**：生产环境通常多集群，无法满足需求。

**解决方向**：
- 集群上下文管理（用户当前在哪集群？）
- 跨集群聚合查询

---

### 6. 可靠性与容错缺失

**现状**：
- 工具执行失败即返回错误
- 无超时控制
- 无重试和降级策略

**影响**：网络抖动、组件临时不可用时用户体验差。

**解决方向**：
- 超时 + 重试 + 熔断
- 降级响应机制
- 健康检查与自动恢复

---

## P2 - 企业级需求

### 7. 响应体验差 — 原始数据直接展示

**现状**：
- 几千行日志直接返回
- 未处理的 JSON 指标数据
- 无结构化输出

**影响**：用户需要自己在海量数据中找答案。

**解决方向**：
- LLM 总结 + 结构化输出（Markdown 表格）
- 自动提取关键信息

---

### 8. 缺乏结果验证与反馈机制

**现状**：
- 执行完就结束，无用户反馈收集
- 错误回答可能被当作正确

**影响**：无法持续优化，用户信任度低。

**解决方向**：
- 👍👎 反馈收集
- 反馈驱动的持续优化
- 不确定时的主动澄清

---

### 9. 技能系统维护性风险

**现状**：
- SKILL.toml 分散在文件系统
- 无版本管理
- 无质量评级

**影响**：技能难以管理，质量不可控。

**解决方向**：
- 技能注册与版本管理
- 技能市场概念
- 质量评级机制

---

### 10. 企业级能力缺失

| 能力 | 现状 | 需求 |
|------|------|------|
| 认证鉴权 | 仅 sender_id | 企业 SSO/LDAP 集成 |
| 访问控制 | 无 RBAC | 按角色/租户隔离 |
| 审计日志 | 基础 interaction 记录 | 完整操作审计 |
| 高可用 | 无 | 多副本 + 负载均衡 |
| 多租户 | 无 | 数据隔离 |

---

## 总结：核心 Gap

| 维度 | 当前（命令路由器） | 目标（AI 运维专家） |
|------|------------------|-------------------|
| 意图理解 | 正则匹配 | 语义理解 + 上下文 |
| 执行模式 | Route → Execute | Plan → Execute → Reason |
| 响应方式 | 原始数据返回 | LLM 总结 + 建议 |
| 交互方式 | 单次查询 | 多轮对话 |
| 错误处理 | 失败即结束 | 重试 + 降级 + 澄清 |
| 学习能力 | 无 | 反馈驱动优化 |

**最关键的 gap 是 AI 能力（理解、推理、生成），这是从「高级脚本管理器」到「AI 运维专家」的本质跨越。**
